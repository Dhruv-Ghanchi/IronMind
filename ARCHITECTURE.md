# ARCHITECTURE.md

## 1. System goal
The system analyzes a polyglot repository and identifies how a change in one layer propagates to other layers.

Target propagation example:
```text
users.email
↓
user_service.py
↓
GET /profile
↓
ProfilePage.tsx
```

---

## 2. Final architecture decisions
### Frontend
- React + Vite
- React Flow for graph visualization

### Backend
- Python + FastAPI

### Graph model
- JSON nodes + edges
- in-memory processing
- structured node model
- rule-based edge generation

### Input mode
- ZIP upload as primary path
- prepared demo repo for judging
- optional user ZIP upload

### Storage model
- temporary extracted files
- in-memory graph / session state
- no persistent database

---

## 3. Layer model
The graph uses 4 fixed architectural layers.

```text
Layer 1 → Database
Layer 2 → Backend
Layer 3 → API
Layer 4 → Frontend
```

### Database layer
Includes:
- tables
- columns
- foreign keys
- views

### Backend layer
Includes:
- python files
- imports
- function definitions
- class/model names
- field references
- backend HTTP client calls

### API layer
Includes:
- GET /route
- POST /route
- PUT /route
- DELETE /route

### Frontend layer
Includes:
- React components
- API calls
- field usage

---

## 4. Supported dependency types
The system detects only these 4 dependency types:

```text
1. Database → Backend
2. Backend → Backend
3. Backend → API
4. API → Frontend
```

These four dependencies are sufficient to demonstrate cross-layer impact propagation without creating graph noise.

## 6. File scanner rules
### Allowed extensions
- `.py`
- `.js`
- `.ts`
- `.jsx`
- `.tsx`
- `.sql`

### Ignored directories
- `node_modules`
- `dist`
- `build`
- `venv`
- `.git`
- `__pycache__`
- `.next`
- `coverage`
- `out`

### Performance budget
- ZIP size ≤ 40 MB
- files scanned ≤ 500
- supported files parsed ≤ 180
- per-file parse timeout ≤ 2 seconds
- total analysis target < 20 seconds
- graceful cutoff at 25 seconds

## 7. Entity extraction model
### SQL entities
- tables
- columns
- foreign keys
- views

### Python entities
- imports
- API routes
- variable / field references
- function definitions
- class / model names
- backend HTTP client calls

### JavaScript / TypeScript entities
- API calls
- field / variable usage
- component names

### API contract entities
- route paths
- HTTP methods
- request body fields
- response fields

---

## 8. Structured node model
All nodes must follow a structured schema.

Example:
```json
{
  "id": "users.email",
  "label": "users.email",
  "type": "column",
  "layer": "database",
  "file": "database/schema.sql",
  "meta": {}
}
```

### Required node fields
- `id`
- `label`
- `type`
- `layer`
- `file`
- `meta`

---

## 9. Edge model
Edges are generated by deterministic rules only.

Example:
```json
{
  "source": "users.email",
  "target": "user_service.py",
  "type": "db_to_backend",
  "confidence": 1.0,
  "reason": "Rule-based field reference match"
}
```

### Required edge fields
- `source`
- `target`
- `type`
- `confidence`
- `reason`

### Edge-generation policy
Rule-based candidate generation only.

No LLM enrichment in graph construction.

---

## 10. Graph layout
Use layered auto-layout in React Flow.

```text
Database → Backend → API → Frontend
```

The graph should render as horizontal columns so the judge can visually understand propagation immediately.

---

## 11. Impact analysis architecture
Impact analysis uses a layered model.

### Step 1 — Impact detection
Use graph traversal (BFS or DFS) to find all downstream impacted nodes.

### Step 2 — Severity scoring
Use a lightweight deterministic score based on:
- impacted node count
- affected layers
- API layer touched
- frontend layer touched
- number of backend services affected
- whether root schema entities changed

### Step 3 — LLM explanation
Use the LLM only to explain the deterministic impact result in natural language.

---

## 12. Natural-language query architecture
Use a hybrid model.

```text
User question
↓
LLM interprets intent
↓
Backend converts to graph query
↓
Graph returns result
↓
LLM explains result
```

The graph remains the source of truth.

---

## 13. Suggested fixes
Use textual recommendations only.

Example:
```text
user_service.py
Update field reference from user.email to user.email_address
```

No code patch generation in main scope.

---

## 14. Fault tolerance / graceful degradation
This is a core requirement.

### Rules
- unsupported files are skipped
- parse failures are skipped
- one bad file must not kill the whole analysis
- partial analysis is always preferable to failure
- raw exceptions are never shown to judges

### User-facing summary example
```text
Partial Analysis Completed
Parsed supported files: 26
Skipped unsupported files: 13
Files with parse issues: 2
```

---

## 15. Hidden debug mode
Debug mode must exist for development and emergency diagnosis.

### Visible only to developers
Expose:
- file scan results
- parse counts
- parse failures
- nodes created
- edges created
- traversal logs
- extraction logs

Default UI must keep debug mode hidden.

---

## 16. Deferred backlog
Not in the 14-hour build unless all core scope is stable.

- patch generation
- PR analyzer
- GitHub URL ingestion as main flow
- frontend → frontend dependency graph
- full function call graph
- full class inheritance graph
- stored procedures / triggers
- graph database support
- deep API contract inference
