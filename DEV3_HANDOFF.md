# Dev 2 to Dev 3: Handoff Notes

These are the integration and architecture notes for Dev 3 (Graph Builder) regarding the entity extraction outputs produced by Dev 2.

## 🏛️ Architecture Overview
Dev 2 operates entirely in-memory using the `code_map` generated by Dev 1. It routes files based on their extensions (`.sql`, `.py`, `.ts`/`.js`/`.tsx`/`.jsx`) to dedicated parsing tools:
- **SQL Extractor:** Safe regex parsing to pull out tables, columns, constraints.
- **Python Extractor:** AST-driven deterministic parsing (handles syntax errors gracefully) capturing imports, functions, classes, HTTP calls, and route decorataors.
- **JS/TS Extractor:** Customized regex parsing mapping React component patterns and frontend REST fetch events.
- **Entity Index Orchestrator:** Unified processing sequence that merges everything into independent deduplicated arrays.

## 🛠️ What Dev 3 Needs to Know

### The Entry Point
- Your entry point is `build_entity_index(SESSION_STORE[id]["code_map"])` (Found in `backend/extraction/entity_index.py`).
- It returns a 12-key `dict`. Node IDs can be taken **directly from the text values**, no mapping or transformation is needed.

### Standardized Formats (Ready for Graph Building)
- All column strings are already explicitly in **`table.column`** format (e.g., `users.email`).
- All route strings are already in **`METHOD /path`** format (e.g., `GET /profile`).
- All `api_calls` strings (frontend) are already formatted to **`fetch /path`** format (e.g., `fetch /profile`).

### The 4 Crucial Edge Rules
When traversing arrays to populate nodes and build edges, these mappings drive relationships:
1. `columns` ↔ `field_refs`
2. `http_calls` ↔ `routes`
3. `file` → `routes`
4. `api_calls` ↔ `routes`

### Layer Assignments (List to Layer Mapping)
The JSON keys generally map out to the 4 primary architectural layers:
- **Database Layer:** `tables`, `columns`, `foreign_keys`, `views`
- **Backend Layer:** `classes`, `functions`, `imports`, `http_calls`, `field_refs`
- **API Layer:** `routes`
- **Frontend Layer:** `components`, `api_calls`, `field_refs`

*(Note: The extraction index DOES NOT store the origin `file` attributing each entity—e.g. `users.email` doesn't inherently point to `schema.sql` inside the return dict. If your Node schema mandates a `"file"` field attached to nodes, please sync with Dev 2 to append this metadata during the traversal.)*

### ⚠️ Known Heuristics & Limitations
- **`field_refs` Heuristic Warning:** Matching Database to Backend edges via `field_refs` is heuristic. It only currently works seamlessly if the Python/JS variable is explicitly named `user` or `data.user` (which automatically coerces to `users.x`). If a codebase utilizes something like `profile.email` or `u.email`, this will be a **silent miss**. Handle this gracefully in your graph generation if an edge doesn't resolve.

### Demo Chain Ground Truth
- The `test_integration.py` script (run via `python -m unittest tests.test_integration` in `/backend`) directly simulates the Dev 1 payload across actual demo repo source scripts. It explicitly **prints the exact output JSON** to the console. Please use this printed JSON as your primary Ground Truth when debugging any missing edges in the graph generator!
